# Tala Backend

Tala is the FastAPI backend for the style-adaptive WhatsApp capstone project.
It verifies WhatsApp webhook requests, receives incoming text messages, applies
style-adaptive response generation, and exposes Personal Context Memory
rule-management endpoints backed by Supabase.

## Project Structure

```text
Backend/
  app/
    main.py                     FastAPI app and WhatsApp webhook handlers
    style_engine.py             Style-aware response pipeline
    personal_context_routes.py  Personal Context Memory API routes
    personal_context_service.py Personal Context Memory service/evaluation logic
    supabase_client.py          Shared Supabase client
    prompt_templates.py         Style prompt templates
```

```text

```

`POST /webhook` receives WhatsApp text messages, generates a style-aware reply,
evaluates personal context rules, and forwards the result to n8n. The forwarded
payload includes:

```json
{
  "message": "are you coming today?",
  "phone": "15551234567",
  "reply": "...",
  "style_mode": "global_contact",
  "profile_contact": "friend",
  "personal_context": {
    "decision": "require_approval",
    "matched_rules": [],
    "reason": "..."
  }
}
```

## Personal Context Memory

Personal Context Memory stores user preferences and behavior rules in Supabase.
The response pipeline evaluates active rules before returning the generated
reply. If Supabase is not configured, style adaptation still works and the
response includes a `personal_context.reason` explaining that rules were
unavailable.

### Supabase SQL

Run the migration in `Backend/supabase/migrations/001_personal_context_memory.sql`.
It creates the rule table, approval-request table, and temporary user-status
table:

```sql
create table if not exists personal_context_rules (
  id bigint generated by default as identity primary key,
  user_id text not null,
  rule_name text not null,
  rule_type text not null,
  rule_value jsonb not null,
  contact_id text,
  topic text,
  action text,
  is_active boolean not null default true,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists approval_requests (
  id bigint generated by default as identity primary key,
  user_id text not null,
  contact_id text,
  original_message text not null,
  generated_reply text not null,
  decision text not null,
  status text not null default 'pending'
    check (status in ('pending', 'approved', 'rejected')),
  reason text,
  matched_rules jsonb not null default '[]'::jsonb,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists user_statuses (
  id bigint generated by default as identity primary key,
  user_id text not null,
  status text not null default 'available',
  status_reason text,
  expires_at timestamptz,
  is_active boolean not null default true,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index if not exists personal_context_rules_user_idx
  on personal_context_rules (user_id);

create index if not exists personal_context_rules_active_idx
  on personal_context_rules (is_active);

create or replace function set_updated_at()
returns trigger as $$
begin
  new.updated_at = now();
  return new;
end;
$$ language plpgsql;

drop trigger if exists personal_context_rules_updated_at
  on personal_context_rules;

create trigger personal_context_rules_updated_at
before update on personal_context_rules
for each row execute function set_updated_at();
```

### Example Rule Requests

Mom always requires approval:

```bash
curl -X POST http://127.0.0.1:8000/rules \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": "default_user",
    "rule_name": "Mom requires approval",
    "rule_type": "contact_requires_approval",
    "rule_value": {"decision": "require_approval"},
    "contact_id": "mom",
    "is_active": true
  }'
```

Money-related topics cannot auto-send:

```json
{
  "user_id": "default_user",
  "rule_name": "Money requires approval",
  "rule_type": "money_requires_approval",
  "rule_value": {"decision": "require_approval"},
  "topic": "money",
  "is_active": true
}
```

During work hours create drafts only:

```json
{
  "user_id": "default_user",
  "rule_name": "Work hours draft only",
  "rule_type": "work_hours_draft",
  "rule_value": "09:00-17:00",
  "is_active": true
}
```

Busy status disables automatic replies:

```json
{
  "user_id": "default_user",
  "rule_name": "Busy means defer",
  "rule_type": "busy_status",
  "rule_value": {"status": "busy"},
  "is_active": true
}
```

Set a temporary user status:

```bash
curl -X POST http://127.0.0.1:8000/personal-context/status \
  -H "Content-Type: application/json" \
  -d '{
    "user_id": "default_user",
    "status": "busy",
    "status_reason": "In a meeting",
    "expires_at": "2026-05-21T18:00:00Z"
  }'
```

Supported statuses:

```text
available
busy
unavailable
traveling
in_meeting
do_not_disturb
```

Temporary status is added to message governance as `user_status` and
`availability`. Rules such as `busy_status` and `availability` can match it and
return `defer`, `require_approval`, or another governance decision.

### Evaluation Output

`evaluate_personal_context_rules(message_data, rules)` returns:

```json
{
  "decision": "require_approval",
  "matched_rules": [
    {
      "id": 1,
      "rule_name": "Mom requires approval",
      "rule_type": "contact_requires_approval",
      "decision": "require_approval"
    }
  ],
  "reason": "Matched personal context rules."
}
```

Possible decisions:

```text
auto_reply
draft_only
require_approval
defer
```

n8n or a future frontend settings page can use this decision to send
automatically, create a draft, request approval, or defer the reply.

The webhook response also includes flattened PCM metadata:

```json
{
  "generated_reply": "...",
  "pcm_decision": "require_approval",
  "matched_rules": [],
  "pcm_reason": "...",
  "final_action": "approval_required",
  "current_status": {
    "status": "busy",
    "status_reason": "In a meeting"
  }
}
```

Only `final_action: "send"` is forwarded to the automatic n8n workflow.
`draft`, `approval_required`, and `deferred` are not sent automatically.

## Testing With a Public URL

Meta needs a public HTTPS URL for webhook callbacks. During local development,
you can expose your local server with a tunneling tool such as ngrok:

```bash
ngrok http 8000
```

Use the generated HTTPS URL as the callback URL in Meta, with `/webhook`
appended:

```text
https://your-ngrok-url.ngrok-free.app/webhook
```

## Notes

- Keep `.env` files, Supabase keys, and WhatsApp access tokens out of version
  control.
- `contact_map.json` should stay local because it can contain real phone
  numbers.
- The app uses Meta Graph API version `v25.0`.
